# Agent Engine

Движок для создания агентов на основе графа состояний. Построен поверх LangChain и LangGraph.

**Для кого:** для коллег, которые хотят создать своего агента, не вникая в детали LangGraph. Достаточно описать состояния, инструменты и промпты — движок собирает граф автоматически.

## 1. Концепция движка

### Агент как граф состояний

Агент — это **граф состояний**. Каждое состояние (`State`) — отдельный «режим работы» с собственным промптом и набором инструментов. Переход между состояниями происходит через специальный инструмент `transition`, который LLM вызывает сам, когда решает, что задача в текущем состоянии выполнена. Это позволяет определять агену разные роли и контроливать промежуточные шаги между состояниями.

Движок автоматически:
- добавляет инструмент `transition` в каждое состояние;
- собирает граф из декларативного описания;
- валидирует конфигурацию до запуска (несуществующие переходы, отсутствующие тулы и т.д.).

### Суммаризация между состояниями

При переходе из одного состояния в другое агент передаёт **краткое резюме** (`summary`) — что было сделано и что нужно сделать дальше. Это резюме движок автоматически подставляет в контекст следующего состояния как сообщение `HumanMessage`. Благодаря этому новое состояние не видит всю историю диалога предыдущего — только итог.

```python
transition(
    next_state="summarize",
    summary="Вычислено среднее = 48.0. Сохранено в memory['result']. Нужно оформить ответ.",
    reasoning="Вычисления завершены, переходим к оформлению."
)
```

### Память как блокнот агента

`memory` — это главная память агента. **key-value хранилище** (`_memory_store`). Оно выполняет три роли:

| Роль | Описание |
|---|---|
| **Блокнот** | Агент записывает промежуточные результаты и читает их в других состояниях |
| **To-do лист** | Агент фиксирует, что нужно сделать следующему состоянию, фиксирует для себя, что уже сделано, что бы не делать повторно |
| **Канал связи** | В мультиагентном режиме агенты делят общую память: один записал — другой прочитал |

```
[state A] — memory(save, key="result") → [state B] — memory(get, key="result") → "42"
[supervisor] — call_agent("math_agent") → память пополнена — [aggregate] — memory(get, key="result")
```

## 3. Установка и настройка

### Установка зависимостей

```bash
pip install -r requirements.txt (рекомендуется через uv)
```

### Настройка бэкенда в `config.yaml`

Движок поддерживает два LLM-бэкенда. Выбери нужный:

```yaml
active_backend: "lmstudio"   # или "gigachat"
```
Тут же можнго настроить параметры подключения, например температуру.

**LM Studio** (локальная модель):
```yaml
backends:
  lmstudio:
    base_url: "http://localhost:1234/v1"
    model: "openai/gpt-oss-20b"
    temperature: 0.3
    timeout: 120
```
Запусти LM Studio, загрузи модель и убедись, что сервер стартовал на `localhost:1234`.

**GigaChat**:
```yaml
backends:
  gigachat:
    model: "GigaChat-2-MAX"
    temperature: 0.2
    timeout: 120
    env_vars:
      base_url: "GIGACHAT_API_URL"
      access_token: "JPY_API_TOKEN"
```
Установи переменные окружения `GIGACHAT_API_URL` и `JPY_API_TOKEN` в своём окружении.

## 4. Готовые шаблоны

Все агенты в `src/agents/` — это **рабочие примеры** с демо-блоками кода. Их можно копировать как основу для своего агента. Каждый показывает конкретный паттерн, а в ноутбуке рядом есть демо-вызовы для ручной проверки.

### `test_agent` — `[work] → END`

Минимальный рабочий пример. Отправная точка, если нужно создать нового агента с нуля: один `entry_point`, одно состояние, базовые тулы.

### `my_agent` — `[work] → [summarize] → END`

Демонстрирует линейный переход между двумя состояниями: `work` выполняет задачу, `summarize` подводит итог. Показывает:
- кастомный тул `plot_chart` в отдельном `tools.py` — тул виден только этому агенту;
- вывод картинки в ленту Streamlit через функцию `ui_image`;
- использование `ask_human` для уточнения данных у пользователя.

### `router_agent` — `[classify] → [math | text | error] → END`

Демонстрирует ветвление: состояние `classify` определяет тип запроса, сохраняет его в память, и агент идёт по одной из трёх веток. Показывает:
- как передавать данные между состояниями через память;
- `memory_injections` с `if_exists`/`if_missing` — контекст подмешивается в состояния автоматически при входе.

### `supervisor_agent` — `[delegate] → [aggregate] → END`

Демонстрирует мультиагентную систему: супервизор вызывает суб-агентов через тул `call_agent` и собирает результаты. Показывает:
- декларативную авторегистрацию суб-агентов через `sub_agents`;
- защиту от повторных вызовов через проверку памяти перед каждым `call_agent`.

## 5. Инструменты (Tools)

### Встроенные shared-инструменты

Доступны во всех агентах без дополнительных действий:

| Инструмент | Назначение |
|---|---|
| `calculator` | Вычисляет математические выражения, hello world в мире тулов. Демонстрирует ручное сохранение информации в памяти агентов |
| `memory` | Сохраняет и читает данные из памяти сессии, агент может сохранять и получать информацию списком за один вызов |
| `summarize` | Формирует структурированное саммари из памяти, можно дать агенту для сжимания контекстного окна придолгой работе внутри одного состояния |
| `think` | Дает функцию размышлений етм ЛЛМ, которые не поддерживают размышления, можно просить ЛЛМ "подумать" местами |
| `ask_human` | Базовый тул для общения с людьми, без него ЛЛМ не сможет что-то спросить у человека |
| `call_agent` | Вызывает другого зарегистрированного агента (мультиагент) |
| `transition` | Переход в другое состояние (добавляется движком автоматически) |

### Создание кастомного инструмента

Создай файл `src/agents/<имя_агента>/tools.py` и экспортируй список `TOOLS`, что бы движок потом автоматически собрал тулы агенту:

```python
from langchain.tools import tool

@tool
def my_tool(param: str) -> str:
    """Краткое описание: что делает инструмент и когда его вызывать.

    Описание критично — Декоратор "@tool" подает описание агенту и он принимает решение о вызове на основе этого описания.
    Чем конкретнее описание, тем точнее агент знает когда и как использовать тул. Приветствуются примеры вызова тулы.

    Args:
        param: Описание параметра — что сюда передавать.

    Returns:
        Описание возвращаемого значения.
    """
    result = ...  # логика инструмента
    return f"Результат: {result}"


TOOLS = [my_tool]
```

Движок подхватит `tools.py` автоматически при вызове `get_tools_dict(agent_name="my_agent")`.

> **Важно про описание тула:** агент видит только `docstring`. Напиши в нём не только «что делает», но и «когда вызывать» и «что передавать в параметры». Плохое описание → агент вызывает тул не вовремя или с неправильными аргументами.

### Тестирование инструментов вручную

Инструменты — это обычные объекты с методом `.invoke()`. Тестируй их отдельно от агента прямо в ноутбуке. Если инструмент работает через invoke, то он будет работать и у агента:

```python
from src.agents.my_agent.tools import plot_chart

# Ручной вызов без запуска агента
result = plot_chart.invoke({
    "title": "Продажи",
    "labels": "Q1,Q2,Q3,Q4",
    "values": "120,95,140,200",
    "chart_type": "bar"
})
print(result)
```

Это позволяет убедиться, что тул работает корректно, до того как подключать его к агенту.

### Искусственное сохранение в память внутри тула

Из любого инструмента можно напрямую писать в память агента, вызывая `memory.invoke(...)`:

```python
from src.tools.tools import memory

@tool
def fetch_data(source: str) -> str:
    """Загружает данные и сохраняет их в память агента."""
    data = ...  # загрузка данных
    # Сохраняем результат как артефакт работы — агент сможет прочитать его
    memory.invoke({"action": "save", "key": "fetched_data", "value": str(data)})
    return f"Данные загружены и сохранены в память под ключом 'fetched_data'"
```

Можно попросить агента что-то сохранить, но это лишние вызовы к LLM, плюс агент может забыть сохранить или сохранить с нарушив формат. Ручное сохранение удобно, когда нужно передать большой объём данных без явного вызова `memory` агентом, работает гарантированно, быстро, в нужном вам формате.

### Вывод сообщений и изображений в Streamlit

Из инструмента можно напрямую отправлять контент в ленту событий Streamlit — без дополнительных ходов агента. Эти функции не только сохраняют текст в логах, а изображение в папке, но и трансилруют результат в стримлит.

**Текст:**
```python
from src.tools.tools import ui_print

ui_print("Промежуточный результат: данные загружены")
```

**Изображение** (сначала сохрани файл, потом вызови):
```python
from src.tools.tools import ui_image
import matplotlib.pyplot as plt

fig, ax = plt.subplots()
ax.bar(["A", "B", "C"], [10, 20, 15])
fig.savefig("/abs/path/to/chart.png")
plt.close(fig)

ui_image("/abs/path/to/chart.png", caption="Результаты")
```

Оба вызова безопасны вне UI-режима: в терминале `ui_print` просто печатает в терминал, `ui_image` ничего не делает.

> **Важно для библиотек с глобальным состоянием** (matplotlib, PIL и подобные): `import` и настройку бэкенда (например `matplotlib.use("Agg")`) делай на уровне модуля, а не внутри тул-функции. Если импорт находится внутри функции, при повторных вызовах библиотека может сбрасывать своё внутреннее состояние — в случае matplotlib это приводит к сохранению пустого белого файла вместо графика.
>
> ```python
> # Правильно — на уровне модуля:
> import matplotlib
> matplotlib.use("Agg")
> import matplotlib.pyplot as plt
>
> @tool
> def plot_chart(...) -> str:
>     fig, ax = plt.subplots()
>     ...
> ```

## 6. Память (Memory)

### Операции с памятью

Тул `memory` поддерживает три действия:

```python
# Сохранить одно значение
memory.invoke({"action": "save", "key": "result", "value": "42"})

# Сохранить несколько за один вызов (экономит раунды LLM)
memory.invoke({
    "action": "save",
    "keys": ["case_id", "status", "result"],
    "values": ["42", "ready", "148.5"]
})

# Прочитать одно значение
memory.invoke({"action": "get", "key": "result"})

# Прочитать несколько за один вызов
got = memory.invoke({"action": "get", "keys": ["case_id", "result"]})
# got["entries"]["case_id"] → "42"

# Посмотреть все ключи
memory.invoke({"action": "list"})
```

### Инъекция памяти в состояние (`memory_injections`)

Вместо того чтобы тратить раунд LLM на вызов `memory(get)`, можно **заранее** объявить, какие ключи подмешать в контекст при входе в состояние. Движок сам читает память и добавляет нужные сообщения в промт модели.

```python
State(
    name="math",
    tools=["calculator", "memory"],
    prompt="...",
    transitions=["END"],
    memory_injections=[
        # Только ключ — добавится как "key: <значение>"
        "calculator_last",

        # Кортеж (ключ, текст_если_есть) — префикс перед значением. Если ключ есть в памяти, то будет добавлен нужный префикс. Например если есть результат работы, можно сделать префикс "этот шаг работы уже сделан: ", это критически важно при повторных заходах модели в состояние, что бы модель не думала, что она в этом состоянии впервые и знала, что часть работы уже проделана
        ("request_type", "Тип запроса: "),

        # Кортеж (ключ, текст_если_есть, текст_если_нет) — что показать если ключа нет, полезно напомнить модели о не проделанной работе. 
        ("result", "Предыдущий результат: ", "Результатов пока нет."),

        # Явный объект MemoryInjection
        MemoryInjection(
            key="error_log",
            if_exists="Зафиксированные ошибки: ",
            if_missing=None,   # не показывать ничего если ключа нет
        ),
    ]
)
```

**Форматы `memory_injections`:** Есть несколько способов как в графе агента указать какие данные из памяти нужно добавить в промт, строка, кортеж из 2 или 3 элементов, словарь, или `MemoryInjection(...)`. Выбирайте удобный себе.

### Практические правила

- Перед новым запуском всегда вызывай `reset_memory()` — иначе данные от предыдущего сценария попадут в новый.
- Договорись о **стабильных именах ключей** между состояниями и агентами (`request_type`, `result`, `case_id`...).
- Используй пакетные операции `memory(save/get)` вместо нескольких отдельных вызовов — это сокращает число раундов LLM.
- В мультиагентном режиме агенты делят одну память. Убедись, что имена ключей не конфликтуют. В системном промте явно прописывай названия ключей, что бы LLM не фантазировала их.

## 7. Создание агента

### Минимальная структура

Создай папку `src/agents/my_new_agent/` с файлами `__init__.py` и `agent.py`:

```python
# src/agents/my_new_agent/agent.py
from src.agent_engine import AgentConfig, State


class MyNewAgent(AgentConfig):
    """Граф: [work] → END"""

    entry_point = "work"

    states = [
        State(
            name="work",
            tools=["calculator", "memory", "think"],
            prompt="""Ты помощник для вычислений.

Алгоритм:
1. Проанализируй запрос.
2. Выполни вычисления через calculator.
3. Сохрани результат: memory(action="save", key="result", value="<ответ>").
4. Вызови transition(next_state="END", summary="Результат: <значение>. Ключ: result.").
""",
            transitions=["END"],
            description="Единственное рабочее состояние",
        )
    ]
```

```python
# src/agents/my_new_agent/__init__.py
from .agent import MyNewAgent
```

### Параметры `State`

| Параметр | Обязательный | Описание |
|---|---|---|
| `name` | да | Уникальное имя состояния |
| `tools` | да | Список имён инструментов |
| `prompt` | да | Системный промпт LLM для этого состояния |
| `transitions` | да | Список допустимых переходов (имена других состояний + `"END"`). Значение `"stay"` доступно всегда и не указывается явно — оно позволяет агенту остаться в текущем состоянии для повторного прохода LLM (например, если нужно уточнить данные или продолжить работу, завершив слишком длинную цепочку langchain) |
| `description` | нет | Описание для документации и `visualize()` |
| `memory_injections` | нет | Ключи из памяти, подмешиваемые при входе в состояние |
| `on_enter` | нет | Функция, вызываемая при входе в состояние |
| `on_exit` | нет | Функция, вызываемая при выходе из состояния |

### Хуки `on_enter` и `on_exit`

Хуки — это обычные Python-функции, которые принимают и возвращают `dict` состояния графа. Используй их для операций, которые должны происходить автоматически, независимо от поведения LLM, при входе или выходе из состояний. Рекомендуется для контроля работы агента, валидации, подготовки данных, можно править память агента.

```python
def prepare_context(state_data: dict) -> dict:
    """Выполняется при каждом входе в состояние."""
    from src.tools.tools import memory
    # Например: автоматически читаем данные из внешнего источника
    memory.invoke({"action": "save", "key": "timestamp", "value": "2026-04-02"})
    return state_data  # обязательно вернуть state_data

def cleanup(state_data: dict) -> dict:
    """Выполняется при каждом выходе из состояния."""
    # Например: валидация результата, логирование, очистка временных ключей
    print(f"Выходим из состояния, память: {state_data.get('memory', {})}")
    return state_data

State(
    name="work",
    tools=["calculator"],
    prompt="...",
    transitions=["END"],
    on_enter=prepare_context,
    on_exit=cleanup,
)
```

### Переходы: как правильно объяснить агенту

Движок автоматически добавляет инструкцию по переходам в конец промпта. прописывает возможные состояния. Когда LLM не правильно вызывает переход между состояниями, то движок пытается угадать состояние, которое указывала модель, путем парсинга текста. Если парсинг не удается, то движок повторяет LLM правила перехода в новое состояние. Тем не менее в самом промпте должны быть чёткие, **декларативные** инструкции: что сделать и в каком состоянии и на каких условиях вызываь `transition`. Это уменьшает количество неудачных попыток перехода, ускоряет работу и минимизирует вызовы к LLM.

**Плохой промпт** (агент не знает когда переходить):
```
Ты помощник. Помогай пользователю с вычислениями.
```

**Хороший промпт** (чёткий алгоритм с явным завершением):
```
Ты помощник для математических вычислений.

Алгоритм:
1. Проанализируй запрос.
2. Используй calculator для вычислений.
3. Сохрани результат в память: memory(action="save", key="result", value="<ответ>").
4. Вызови transition(next_state="summarize", summary="Вычислено <результат>. Ключ: result.").

ВАЖНО: шаг 4 обязателен. Не завершай ответ без вызова transition.
```

**Рекомендации по промпту:**
- Перечисли шаги явно и пронумеруй их.
- В последнем шаге всегда явно требуй вызов `transition` с конкретными состоянми.
- Добавь `ВАЖНО: вызов transition обязателен` — это снижает вероятность «забытого» перехода.
- Если состояний несколько, опиши **условие** перехода в каждое: «если X — переходи в A, если Y — переходи в B».

### Мультиагентная система

Для запуска под-агентов объяви `sub_agents` — движок автоматически их зарегистрирует. Тула для вызова агента принимает первый промт, будто его пишет челвоек и возвращет тоже текст. Для обмена более сложной информацией, используйте memmory, если нужно передавать файлы, то передавайте пути к файлам. Мультиагентная система бесполезна без возможности паралельного вызова к LLM, так как состояния и так уже меняют роль, инструкции и набор тулов. Однако это удобно для командной работы и когда суб агент имеет смысл на существование отдельно от всей системы:

```python
class SupervisorAgent(AgentConfig):
    entry_point = "delegate"
    sub_agents = ["test_agent", "router_agent"]  # должны существовать в src/agents/

    states = [
        State(
            name="delegate",
            tools=["call_agent", "memory", "think"],
            prompt="""Координируй агентов.
1. Вызови нужного агента: call_agent(agent_name="test_agent", query="...").
2. Сохрани результат в память.
3. Вызови transition(next_state="aggregate", summary="Делегирование завершено. Ключ: result.").
""",
            transitions=["aggregate"],
        ),
        ...
    ]
```

Ограничения `sub_agents`:
- Все имена должны существовать в `src/agents/`.
- Нельзя добавлять себя в `sub_agents`.
- Ручной `register_agent(...)` для `sub_agents` не нужен.

## 9. Логирование

По умолчанию логи выводятся в **агрегированном режиме**: часть истории сообщений, которая не изменилась между LLM-запросами, не дублируется. В консоль выводится только **дельта** — новые сообщения с момента последнего запроса. Это упрощает читаемость логов, но можно включить сохранение полной истории запросов.

статистика для подсчета ресурсов, показывает количество вызовов LLM, длину истории в mesages, количество входящих токенов запроса и количество исходящих токенов запроса: `REQ N | msgs=<кол-во> in=<токены> out=<токены>`

Большое количество вопросов сигнализирует о долгой работе графа, длинная история в запросе сигнализирует о возможном переполнении контекста, так же большое кол-во вызова тулов в истории может ломать LLM и заставляет их галюцинировать, количество токенов тоже сигнализирует о взможном переполнении контекстного окна, потому что последний печатаемый токен все еще базируется на основе всех токенов перед ним. Слишком длинный контекст может приводить к проблеме "lost in the midle", к потере внимания к некоторым деталям в запросе. Желательно минимизировать эти цифры, это удешевляет агентов, ускоряет их работу и делает точнее их ответы.

### Настройка уровня детализации

```yaml
logging:
  level: detailed      # off | simple | detailed (регулирует количество текста в логах)
  aggregated: true     # true — агрегированный вывод, false — полная история
  renderer: auto       # auto | rich | ansi | off (включает цветовое декорирование логов, что бы отличать тип сообщений в логах друг от друга)
```

### Сырые запросы и ответы LLM

Если нужно видеть полный JSON запроса/ответа к LLM (например, при отладке странного поведения):

```yaml
logging:
  raw_io: true
```

| `streamlit.refresh_feed` | `"0.5s"`, `"1s"` | Частота обновления ленты событий. Если интерфейс лагает в браузере — увеличь значение (`"1s"` или `"2s"`) |

## 11. Сессии и чекпоинты

Движок автоматически сохраняет полный `AgentState` после каждого шага агента в SQLite-базу. Это позволяет восстановить работу после падения, загрузиться с любой точки истории и тестировать агента с разными параметрами.

### Как включить

В `config.yaml`:

```yaml
sessions:
  enabled: true
  db_path: "sessions/checkpoints.db"     # SQLite с состояниями
  registry_path: "sessions/sessions.json" # реестр сессий и имён
  max_sessions_per_agent: 20              # 0 = безлимит
```

При загрузке агента старые сессии сверх лимита удаляются автоматически. Сессии с именованными чекпоинтами защищены от удаления.

### Два типа чекпоинтов

**Авто-чекпоинты** — создаются LangGraph после каждого node. Хранятся до лимита `max_sessions_per_agent`.

**Именованные чекпоинты** — пользователь тегирует любой авто-чекпоинт. Защищены от авто-удаления, живут пока не удалишь вручную.

### API агента

```python
# Новая сессия создаётся автоматически
result = agent.invoke(["Задача..."])
session_id = agent.last_session_id

# Продолжить ту же сессию (добавить сообщение)
result = agent.invoke(["Уточнение..."], session_id=session_id)

# Восстановить после падения (resume с последнего checkpoint)
result = agent.resume(session_id)

# Просмотр истории чекпоинтов
for cp in agent.list_checkpoints(session_id):
    print(cp["state_name"], cp["timestamp"], cp.get("name", ""))

# Тегировать последний чекпоинт
agent.tag_checkpoint(session_id, "baseline_v1", note="до смены схемы")

# Посмотреть состояние в точке чекпоинта
state = agent.get_checkpoint_state("baseline_v1")
print(state["memory"])

# Форк — новая независимая сессия из именованного чекпоинта
new_sid = agent.fork("baseline_v1")
result = agent.invoke(["Повтори"], session_id=new_sid)

# Форк с правками — изменить память перед запуском
new_sid = agent.fork(
    "baseline_v1",
    edits={"memory": {"db_schema": "orders_v2"}},
    description="тест с новой схемой",
)
result = agent.invoke(["Повтори анализ"], session_id=new_sid)
```

## 12. Частые ошибки

### Агент не вызвал `transition` и завис

**Симптом:** агент даёт ответ, но остаётся в том же состоянии (`stay`) — в логах повторяется одно и то же состояние.

**Причина:** агент завершил ответ, не вызвав `transition`. Движок интерпретирует это как «остаться в текущем состоянии» и запускает его снова.

**Решение:** добавь явное и декларативное указание в конец промпта:

```
# Было (агент может «забыть»):
"Выполни запрос пользователя и дай ответ."

# Стало (явный обязательный последний шаг):
"4. После ответа ОБЯЗАТЕЛЬНО вызови transition(next_state='END', summary='...').
ВАЖНО: ответ без вызова transition = ошибка."
```

---

### Агент зациклился и не переходит в новое состояние

**Симптом:** агент повторно входит в одно и то же состояние несколько раз без прогресса.

**Причина 1:** агент вызывает `transition(next_state="stay", ...)`, считая, что работа не закончена, хотя данных достаточно.

**Решение:** уточни **критерий завершения** в промпте:
```
"Если в памяти есть ключ 'result' — работа завершена, вызывай transition(next_state='END').
Не вызывай stay если result уже сохранён."
```

**Причина 2:** агент не находит нужные данные в памяти и запрашивает их снова.

**Решение:** используй `memory_injections`, чтобы нужные ключи автоматически подмешивались при входе в состояние — агент сразу видит контекст и не тратит раунд на `memory(get)`.

**Причина 3:** превышен `recursion_limit` внутри состояния (лимит = 25 шагов на состояние).

**Решение:** сократи количество инструментов в состоянии или раздели логику на несколько состояний.

---

### Тул не найден при запуске агента

**Симптом:** `KeyError` при сборке агента или предупреждение о пустом списке инструментов.

**Причина:** имя тула в `State.tools` не совпадает с именем в `TOOLS`.

**Диагностика:**
```python
tools_dict = get_tools_dict(agent_name="my_agent")
print(list(tools_dict.keys()))  # посмотреть все доступные имена
```

---

### Дублирующиеся имена тулов

**Симптом:** `ValueError: Дублирующееся имя инструмента: <name>`.

**Причина:** два разных тула объявлены с одним именем (имя берётся из имени функции, а не переменной).

**Решение:** убедись, что имена функций-инструментов уникальны во всех `tools.py` файлах проекта.

---

### `entry_point` не совпадает с именем состояния

**Симптом:** `ValueError: entry_point 'work' отсутствует в states`.

**Причина:** опечатка в `entry_point` или в `name` состояния.

**Решение:** `entry_point` должен буквально совпадать с полем `name` одного из состояний в `states`.

---

### Память не сброшена между запусками

**Симптом:** агент в новом запуске «знает» данные из предыдущего.

**Причина:** `_memory_store` — глобальный объект сессии, он не очищается автоматически.

**Решение:** всегда вызывай `reset_memory()` перед новым сценарием:
```python
from src.tools.tools import reset_memory
reset_memory()
agent.invoke(["новый запрос"])
```
